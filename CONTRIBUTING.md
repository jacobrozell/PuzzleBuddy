## Before you start

1. Read [AGENTS.md](AGENTS.md) if you are an automated agent or new contributor.
2. Open an issue for large features or breaking changes so we can align on approach.
3. Fork the repo and create a feature branch from `main` (or `master`).
4. Keep pull requests focused — one logical change per PR is easier to review.

## Development setup

### Xcode project generation

`Puzzle Buddy.xcodeproj` is generated by [XcodeGen](https://github.com/yonaskolb/XcodeGen) from `project.yml` and is **not** committed. After every pull:

```bash
brew install xcodegen   # if needed
xcodegen generate
```

If you add new source files, update `project.yml` (or place files under existing `sources` paths) and regenerate.

### Firebase config

```bash
cp GoogleService-Info.plist.example GoogleService-Info.plist
# Fill in real Firebase values for local runs
./Scripts/install-git-hooks.sh
```

CI uses the example plist with placeholder values. Firebase bootstrap skips configuration when `GOOGLE_APP_ID` contains `REPLACE_WITH`.

### Local verification

```bash
xcodegen generate
swiftlint lint
Scripts/ci/run-tests.sh "platform=iOS Simulator,name=iPhone 16"
```

See [docs/development.md](docs/development.md) for simulator selection, DerivedData tips, and troubleshooting.

## Code style

### SwiftLint

Configuration lives in `.swiftlint.yml`. Notable rules:

- **No `print()` in app code** — custom rule `no_print_statements` warns in `Puzzle Buddy/**/*.swift`. Use `AppLog` instead.
- **Opt-in `force_unwrapping`** — warnings on force unwraps; prefer safe unwrapping or early returns.
- Generous line length (200 warn / 300 error) to match existing style.

Run before committing:

```bash
swiftlint lint
```

### Logging

```swift
// Good
AppLog.shared.info(.puzzles, eventName: "puzzle_added", message: "Puzzle saved.", metadata: ["puzzle_status": status])

// Avoid
print("Saved puzzle")
Analytics.logEvent("custom_event", parameters: ["email": userEmail])  // never log PII
```

### Design tokens

Use `Brand` and `DS` from `DesignTokens.swift` instead of hard-coded colors and spacing:

```swift
// Good
.foregroundStyle(Brand.textPrimary)
.padding(DS.Spacing.s4)
.buttonStyle(BrandPrimaryButtonStyle())

// Avoid
.foregroundColor(.gray)
.padding(16)
```

Add new tokens to `DesignTokens.swift` when a color or spacing value is reused across screens.

### SwiftUI patterns

- Prefer `@MainActor` on `ObservableObject` types that touch UI or SwiftData from the main thread (`PuzzleStore`, `ErrorHandling`).
- Use `@StateObject` for owned store lifetimes; `@EnvironmentObject` for app-wide error handling.
- Keep view files focused — extract reusable subviews (see `HelperViews/` for rating and difficulty controls).

### Naming and organization

| Area | Convention |
|------|------------|
| Views | `*View.swift` in `Views/` or `Login/` (onboarding only) |
| Stores | `*Store.swift` in `Helpers/` |
| Utilities | `Util/` |
| MARK comments | Use `// MARK: - Section` to group extensions and nested types |
| File header | Keep existing copyright block when editing |

Note: `Login/` folder name is legacy — contains `OnboardingView` only.

## Architecture conventions

### Data flow (1.0)

1. `AppShell` → `RootView` → `PuzzleView` (or onboarding first run)
2. `PuzzleStore` loads and saves puzzles via SwiftData (`PuzzleRecord`)
3. Views call `PuzzleStore.add`, `update`, or `delete`
4. Errors surface through `ErrorHandling.handle(error:title:)`

Future account sync: [specs/planned/auth-cloud-sync.md](specs/planned/auth-cloud-sync.md) — not in app today.

### SwiftData records

`PuzzleRecord` mirrors puzzle fields for on-device storage. When adding fields:

1. Update `PuzzleRecord` and its `init(from:)` / `apply(from:)` / `toPuzzle()` methods
2. Add tests in `PuzzlePersistenceTests`

### Dictionary serialization (export / tests)

`Puzzle.getDataFields()` and `fromData(_:)` support export round-trip tests — not cloud sync. When adding fields:

1. Update both methods
2. Add tests in `PuzzleSerializationTests`

### Dependencies

Swift packages in `project.yml`:

- **Firebase** — `FirebaseCore`, `FirebaseAnalytics`, `FirebaseCrashlytics` only
- **SwiftData** (system)

Full reference: [docs/architecture.md](docs/architecture.md), [AGENTS.md](AGENTS.md).

## Logging and analytics

All production analytics go through `AppLog.shared` with an `eventName`. Only allowlisted events and parameter keys are forwarded to Firebase Analytics.

When adding a new measurable action:

1. Pick a snake_case `eventName`.
2. Add it to `PuzzleAnalyticsEventMapping.allowlistedEvents` in `AppLogging.swift`.
3. Use only allowlisted metadata keys — see [docs/telemetry.md](docs/telemetry.md).
4. Never pass email, UID, password, or display name in metadata.
5. Update [docs/telemetry.md](docs/telemetry.md) and tests in `AppLoggingTests`.

Details: [docs/analytics.md](docs/analytics.md) (quick ref) · [docs/telemetry.md](docs/telemetry.md) (full spec).

## Accessibility

New interactive controls should include:

- **`accessibilityLabel`** — concise VoiceOver description
- **`accessibilityIdentifier`** — use constants from `A11yID` in `DesignTokens.swift` for UI tests

```swift
TextField("Email", text: $email)
    .accessibilityLabel("Email address")
    .optionalAccessibilityIdentifier(A11yID.loginEmailField)
```

Respect `@Environment(\.accessibilityReduceMotion)` for animations (see `BrandBackground`).

Track broader WCAG work in [accessibility/accessibility_todo.md](accessibility/accessibility_todo.md). Full criteria mapping: [docs/wcag.md](docs/wcag.md). Update [accessibility/wcag-2.1-aa/conformance-matrix.md](accessibility/wcag-2.1-aa/conformance-matrix.md) when closing accessibility gaps.

## Testing

### Unit tests (`Puzzle BuddyTests`)

- Add tests for serialization, parsing, and pure logic.
- Use `Puzzle.fixture()` helpers for test data.
- Run with ⌘U or `Scripts/ci/run-tests.sh`.

### UI tests (`Puzzle BuddyUITests`)

- Use `A11yID` constants to locate elements.
- Keep launch tests fast; avoid hard dependency on Firebase for smoke tests where possible.

### What to test for your change

| Change type | Expected tests |
|-------------|----------------|
| New `Puzzle` field | Serialization round-trip in `PuzzleSerializationTests` |
| New a11y label contract | `AccessibilityLabelTests` or UI test identifier |
| Auth or store behavior | Unit test with mocks or documented manual test plan in PR |

More detail: [docs/testing.md](docs/testing.md).

## Firebase and secrets

- **Never** commit `GoogleService-Info.plist`. The pre-commit hook rejects staged real plists.
- **Never** commit API keys, service account JSON, or APNs certificates.
- Use `GoogleService-Info.plist.example` as the template; CI copies it automatically.
- Deploy rule changes with `firebase deploy --only firestore:rules` and include rule updates in the same PR when schema or access patterns change.

Firebase setup guide: [docs/firebase-setup.md](docs/firebase-setup.md).

## Pull request checklist

Before requesting review:

- [ ] `xcodegen generate` — project builds locally
- [ ] `swiftlint lint` — no new warnings you introduced
- [ ] Tests pass (⌘U or `Scripts/ci/run-tests.sh`)
- [ ] No `print()` in app target code
- [ ] Analytics events use `AppLog` and allowlist (if applicable)
- [ ] New UI controls have accessibility labels/identifiers where practical
- [ ] No secrets or real `GoogleService-Info.plist` in the diff
- [ ] Documentation updated if you changed setup, architecture, or public behavior
- [ ] [roadmap.md](docs/roadmap.md) updated if you shipped or reprioritized planned product features (stats, model fields, competitive items)

### PR description template

```markdown
## Summary
- What changed and why

## Test plan
- [ ] Built and ran on simulator/device
- [ ] Unit tests pass
- [ ] Manual steps (if any)
```

## CI expectations

Every PR runs:

1. **SwiftLint** on Ubuntu
2. **Build & Test** on `macos-15` with iPhone 16 Simulator

Fix CI failures before merge. Test result bundles are uploaded as artifacts on failure (`test-results`).

Workflow: [.github/workflows/ci.yml](.github/workflows/ci.yml).
