## Phase 1 — Ship-blocker behavioral bugs

### 1a. Edit mode: copy-on-edit + Cancel + refresh-on-save
**Files:** `App/Views/PuzzleViews/PuzzleDetail.swift` (toolbar `:38-78`), `App/Views/PuzzleViews/PuzzleForm.swift:19-25`.

Root cause: `Puzzle` is a `class`, so `PuzzleFormViewModel(puzzle:)` shares the same instance the detail binds to — edits mutate the live model before Save, with no rollback.

1. Add a deep copy on `Puzzle` (`func copy() -> Puzzle`) or a `PuzzleFormViewModel(editing:)` that builds a fresh `Puzzle` from all fields (name, pieces, rating, difficulty, time, dates, status, notes, source, purchase*, metadata enums, dimensions, photos, completions, tags, barcode, isDemo).
2. Entering edit mode builds the VM from a **copy**; original stays intact.
3. Add a **Cancel** button (leading slot while `isEditable`): `editFormVm = nil; isEditable = false`.
4. On **Save**: `try ps.update(puzzle: editFormVm.puzzle)`, then refresh the binding from the store (mirror redo at `:88-90`):
   - `if let refreshed = ps.puzzles.first(where: { $0.id == editFormVm.puzzle.id }) { puzzle = refreshed }`
   - then clear `editFormVm`, `isEditable = false`.

### 1b. Auto-save refresh consistency
**File:** `PuzzleDetail.swift:181-190`. After the progress slider's `ps.update`, refresh the local `puzzle` binding the same way so detail/list never drift.

### 1c. Guard `navigationDestination`
**Files:** `PuzzleList.swift:213-216`, `PickNextPuzzleView.swift:58-61`. Render a graceful "no longer available" fallback (or pop) when the puzzle ID isn't found, instead of an empty view.

**Acceptance:** edit→back reverts nothing and persists nothing; Save updates list+detail together; navigating to a deleted puzzle shows a graceful screen.

## Phase 2 — Data integrity hardening

### 2a. Atomic replace-all restore
**File:** `PuzzleStore.swift:100-152`. Prefer **validate-then-clear**: validate all incoming rows first; only `clearAllPuzzles()` once validation passes. (Fallback: snapshot before clear and restore on fatal error.)

### 2b. Surface save/fetch failures
**File:** `PuzzleStore.swift`.
- `delete(at:)` / `deleteLocally` (`:192-207`): make throwing; propagate save errors to callers (`PuzzleList`, `SettingsView`, `PuzzleView`) to show `eh.handle(...)`.
- `fetchPhotoRecords`/`fetchCompletionRecords`/`fetchRecord` (`:428-461`): log an `AppLog.warning` with a stable event name on `catch` (keep resilient return).
- `updateLocally` (`:324-344`): replace silent `guard … else { return }` with a thrown `PuzzleStoreError.recordNotFound` so a real failure surfaces (today analytics logs success regardless).

### 2c. Visible persistent-store fallback
**File:** `PuzzleModelContainer.swift:36-46`. Keep in-memory fallback but flag it (e.g. `UserPreferences.isRunningInEphemeralStore`), show a first-launch banner ("changes won't be saved"), and log a Crashlytics non-fatal with a stable `eventCode`.

### 2d. Count dropped import rows
**Files:** `PuzzleCollectionJSONImporter.swift:59-61`, `PuzzleBackupNormalizer.swift:133-135`. Stop discarding empty-name rows before the store counts them; fold dropped count into `PuzzleImportSummary.skippedInvalid` and surface in `IPDbImportSummarySheet`.

### 2e. Sparse multi-completion restore
**File:** `PuzzleCollectionJSONImporter.swift:187-192`. When `timesCompleted > 1` and `completions` is empty, synthesize N placeholder completions (1…N).

### 2f. Shopping-mode invalid-barcode feedback
**File:** `ShoppingModeView.swift:117-118`. On normalize failure, fire `BarcodeScanFeedback` error haptic + VoiceOver announcement (match `PuzzleList.swift:696-701`).

### 2g. Loading state (from pre-decided choice)
Wire `PuzzleStore.state` into a real loading indicator in `PuzzleList`.

**Acceptance:** restore failure can't wipe a collection; delete failures reach the user; `imported + skipped* == file rows`; shopping mode gives feedback; list shows a loading state.
